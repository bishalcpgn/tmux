    tmux

    Terminal multiplexer

    One terminal, many windows and panes, sessions that never die when you disconnect. This guide builds the mental model first, then layers on commands, configuration, and plugins.

    1. Why tmux

    Before any commands, a clear picture of what tmux is and the one problem it solves better than anything else: keeping your work alive when your connection to it does not. Read this section slowly; everything later in the guide builds on the model laid out here.

    How to read this guide

    The sections are numbered and meant to be followed in order. Section 1 is the theory. From section 2 onward you are at the keyboard, so keep a terminal open and run each command and lab as you reach it.

    1.1 First, the words: terminal, shell, multiplexer

    Three terms get used loosely and trip up beginners. Pinning them down makes the rest of the guide obvious, because every tmux command acts on one of these layers.

    • Terminal (terminal emulator). The application window you type into, such as GNOME Terminal, iTerm2, Windows Terminal, Kitty, or Ghostty. It draws text and forwards your keystrokes. It does no computing of its own.
    • Shell. The program running inside the terminal that interprets what you type, such as bash, zsh, or fish. When you run ls, the shell is what executes it.
    • Terminal multiplexer. A layer that sits between the two. It runs many shells at once, organises them into windows and panes, and, most importantly, keeps them running even when no terminal is attached. tmux is that layer. "Multiplexer" simply means many into one: many independent shells delivered through one terminal connection.

    The three are layers stacked one behind the next, and a single keystroke travels through all of them. Follow one round trip and each word's job becomes concrete: the terminal forwards what you type, tmux routes it to whichever shell is active, the shell runs it, and the output retraces the same path back to the screen.

    fig: tmux architecture

    1.2 The one idea that matters: detach and re-attach

    tmux is built as a client-server program, and this is the whole reason it exists. When you first run tmux, it starts a long-lived server in the background and your terminal becomes a client attached to it. Your shells run inside the server, not inside your terminal window.

    Because of that split, your terminal is just a viewport you can connect to and disconnect from at will. Detach, and the server keeps every shell running; re-attach later, from the same terminal or over SSH from a different machine, and you are dropped back into exactly the state you left.

    the server lane never breaks. only the client connects and disconnects.

    1.3 tmux vs modern terminals

    In 2026 your terminal emulator can almost certainly already split panes and open tabs. So is tmux still worth learning? Yes, but only once you see that a terminal and a multiplexer solve two different problems. This part draws the line, so you know when to reach for which.

    A terminal's split is a view; a tmux split is the process

    Modern emulators such as WezTerm, Kitty, iTerm2, Windows Terminal, and Ghostty draw tabs and split panes natively, with GPU-accelerated text, ligatures, and inline images. Those splits are local to the GUI: they live in the application running on the machine in front of you, and they disappear when you close the window. (A few terminals, such as Alacritty, deliberately ship no splits at all and expect you to bring a multiplexer.)

    tmux instead runs as a background server. Its windows and panes live inside that server, not in the GUI, so they survive a closed window, and when the server runs on a remote host they survive an SSH disconnect. That is the whole distinction: a terminal's split is a view of a process; a tmux split is the process, one you can walk away from and come back to.

    The remote case is where it clicks

    SSH into a server, start a long build inside tmux, detach, and close your laptop. The build keeps running on the server. Your terminal's own splits cannot do this: if the SSH link drops, anything running remotely without tmux is cut off with it.

    Where tmux wins, where it loses

    What you needTerminal's own splitstmux
    Survive a dropped SSH connection / persist on a remote hostNoYes
    Keep work running after the local window closesNoYes, the server keeps running
    Work over plain SSH on any server (preinstalled or one package away)NoYes
    Script and reopen an entire layout on demandLimitedYes
    GPU rendering, ligatures, inline images (Sixel / Kitty graphics)YesNo: text cells only, and it can get in the way
    Zero setup, on-screen discoverable keysYesNo: keybinding-driven, some config
    System clipboard and mouse that just workYesNeeds a line of config or a plugin

    The rule of thumb: if you never leave one machine and never disconnect, your terminal's own splits are simpler and render better. The moment remote servers, long-running jobs, or flaky connections enter the picture, tmux earns its place, and because it runs inside any terminal, learning it once carries across every machine and emulator you will ever use.

    The 2026 landscape

    tmux is not the only multiplexer, and the boundary with terminals is blurring:

    • GNU screen: the original multiplexer that tmux largely replaced. Still preinstalled on some older systems, but rarely chosen for new setups.
    • Zellij: a modern multiplexer written in Rust, with on-screen keybinding hints, built-in layouts, and session resurrection. Friendlier for newcomers, but not yet as ubiquitous or as commonly preinstalled as tmux.
    • WezTerm: a terminal with its own built-in multiplexer that can persist sessions and attach to remote hosts, blurring the terminal/multiplexer divide.
    • Ghostty: a fast, modern terminal (1.0 landed at the end of 2024) that deliberately leaves multiplexing out and points you at tmux or Zellij for it.

    1.4 The mental model

    Everything in tmux nests. Get this one picture in your head and every command afterward will make sense, because each command operates on one of these four levels.

    server session window pane

    Read it from the outside in:

    • Server. Started the first time you run tmux. Holds everything. You rarely touch it directly.
    • Session. A workspace, usually one per project. This is what you attach to and detach from. Think "dev", "ops", "blog".
    • Window. Like a browser tab inside a session. Full-screen by default. Each window has a number and a name.
    • Pane. A split of a window. One window can be carved into many panes, each running its own shell.

    2. The prefix key

    tmux needs a way to tell its own commands apart from text you are typing into a shell. The answer is the prefix: a key you press first, which puts tmux into "listen" mode for the next single keystroke.

    default prefix is Ctrl+b. throughout this guide it is shown as the green "prefix" key.

    So when you see prefixc it means: press Ctrl+b, let go, then tap c. You do not hold them all down together. Almost every tmux shortcut follows this pattern.

    Remapping the prefix

    Many people find Ctrl+b an awkward stretch and remap the prefix to something easier to reach. You will do exactly that in the configuration section. Until then, this guide uses the default Ctrl+b.

    2.1 Three keys to know before anything else

    KeysAction
    prefix?List every key binding (your built-in cheat sheet). Press q to close.
    prefix:Open the tmux command prompt to type full commands.
    prefixdDetach from the session and drop back to your normal shell.

    Lab 00 The Houdini Session

    Experience the core magic of tmux: detaching and re-attaching a live process.

    1. Open your terminal and start a new tmux session by typing: tmux
    2. Launch a simple task that prints counts continuously: for i in {1..100}; do echo "Count: $i"; sleep 1; done
    3. Press your prefix sequence Ctrl+b then tap d to detach. Your terminal will drop back to your host shell.
    4. List the active sessions running in the background: tmux ls
    5. Re-attach to resume: tmux attach
    Verification Checklist

    Upon re-attaching, the loop should still be printing counts (e.g., Count: 15, Count: 16) without interruption.

    Lab 01 The Nested Scaffold

    Build and navigate the entire tmux hierarchy: Server → Session → Window → Pane.

    1. Create a new session named sandbox: tmux new -s sandbox
    2. Create a second window: press Ctrl+bc
    3. Split this new window into two side-by-side panes: press Ctrl+b%
    4. Identify the structure by looking at the green status bar: note the session name on the left, and the two window indexes in the middle.
    Verification Checklist

    Verify that your status bar shows sandbox as the session, with windows 1:zsh and 2:zsh* (where * denotes the active window containing your two split panes).

    Lab 02 The Command Room

    Practice accessing tmux's internal control center and querying keys.

    1. Inside an active tmux session, open the interactive helper menu: press Ctrl+b?
    2. Scroll through the key bindings using the arrow keys. Press q when done to exit the help menu.
    3. Enter command mode to speak directly to the server: press Ctrl+b:
    4. Type display-message "Hello from the command prompt" and press Enter.
    Verification Checklist

    Your message should flash briefly at the bottom of the screen in the status bar area, confirming tmux executed the command successfully.

    3. Sessions

    A session is your top-level workspace. Most commands here are typed in your normal shell (before attaching) and start with tmux. The shortcuts work once you are inside.

    3.1 The everyday loop: detach, list, attach

    Nearly all session work is one short cycle: leave a session running, see what is alive, then drop back into the one you want. Section 1 covered detaching; the two moves that complete the loop are listing and attaching. Learn these three and you already have the core of tmux.

    you@laptop: ~
    # 1. detach: leave the session running, back to your shell
    # (a key press, not a command: prefix then d)
     
    # 2. list: what is still running?
    you@laptoptmux ls
    dev: 2 windows (created Mon Jun 15 14:30:00 2026)
    ops: 1 windows (created Mon Jun 15 14:32:10 2026)
     
    # 3. attach: jump back into the one you want, by name
    you@laptoptmux attach -t dev
    # short form: tmux a -t dev
     

    The same loop also covers an involuntary detach. If your SSH connection drops or your terminal closes, the session is just left running on the server, so tmux ls followed by tmux attach -t dev drops you back exactly where you were. The tables below break each step out in full.

    3.2 From your shell

    CommandAction
    tmuxStart a new, unnamed session and attach to it.
    tmux new -s devStart a new session named dev.
    tmux lsList all running sessions.
    tmux attach -t devAttach to the session named dev (short: tmux a -t dev).
    tmux new -s dev -dCreate dev in the background without attaching.
    tmux kill-session -t devStop the dev session.
    tmux kill-serverStop everything. Every session gone.

    3.3 From inside tmux

    KeysAction
    prefixdDetach (the session keeps running).
    prefixsShow an interactive tree of all sessions and switch.
    prefix$Rename the current session.
    prefix( prefix)Jump to the previous, or next, session.
    Workflow

    Give every project its own named session. Then tmux a -t blog drops you straight into that world. prefixs is the fast way to hop between them without leaving tmux.

    Lab 03 The Workspace Hopper

    Learn to manage, swap, and cleanup multiple project workspaces.

    1. Detach from your current session: press Ctrl+bd
    2. Create a new session in the background named ops: tmux new -s ops -d
    3. Create another session and attach to it: tmux new -s dev
    4. While inside dev, list all sessions and switch to ops: press Ctrl+bs, select ops using arrow keys, and press Enter.
    5. Rename the ops session to production: press Ctrl+b$, clear the line, type production, and hit Enter.
    Verification Checklist

    Press Ctrl+bs again. You should see dev and production listed. Press q to dismiss the list.

    4. Windows

    Windows are the tabs of a session. Each one fills the whole screen and shows up in the status bar as number:name. The active window is marked with an asterisk.

    KeysAction
    prefixcCreate a new window.
    prefix,Rename the current window.
    prefixn / prefixpNext / previous window
    prefix0 … prefix9Jump straight to window by number.
    prefixwInteractive list of windows across sessions.
    prefixfFind a window by searching its text.
    prefix&Kill the current window (asks to confirm).
    prefix.Move the window to a different number.
    Reading the status bar

    1:node* means window 1, named "node", currently active. A - marks the last window you were on, so prefixl toggles between the two.

    Lab 04 The Tab Architect

    Organize multiple full-screen tasks inside a single session.

    1. Create three new windows in your session: press Ctrl+bc three times.
    2. Go to the second window by pressing Ctrl+b2.
    3. Rename this window to database: press Ctrl+b,, type database, and press Enter.
    4. Switch back and forth between the last two active windows using the quick-swap key: press Ctrl+bl.
    5. Close the current window: press Ctrl+b& and type y to confirm.
    Verification Checklist

    Check the status bar. The window list should no longer show the closed window, and the active window index indicator should update immediately.

    5. Panes

    Panes split a single window into multiple shells you can see at once. This is where tmux feels powerful: editor on the left, server logs top-right, a free shell bottom-right, all in one view.

    the % vs " confusion, solved: % makes a vertical divider (side by side), " makes a horizontal divider (stacked).
    The one thing beginners mix up

    % is a vertical bar, so it splits into left and right. " looks like it lies flat, so it splits into top and bottom. If a remembered shortcut feels backwards, this is almost always why.

    5.1 Creating and moving between panes

    KeysAction
    prefix%Split into left and right.
    prefix"Split into top and bottom.
    prefix←↑↓→Move to the pane in that direction.
    prefixoCycle through panes in order.
    prefixqFlash pane numbers; press a number to jump to it.
    prefix;Toggle to the last active pane.

    5.2 Resizing, zooming, and rearranging

    KeysAction
    prefixzZoom the current pane to fullscreen, then again to restore. Huge for focus.
    prefixCtrl ←↑↓→Resize the pane (hold and tap the arrow repeatedly).
    prefixSpaceCycle through preset layouts (even-horizontal, tiled, and more).
    prefix{ prefix}Swap the pane with the one before, or after, it.
    prefix!Break the current pane out into its own window.
    prefixxClose the current pane (asks to confirm).

    Lab 05 The Quad Split

    Master splits, zooming, and layouts to maximize your screen real estate.

    1. Clear your current window's extra panes if any, or open a fresh window (Ctrl+bc).
    2. Split the window vertically (side-by-side): press Ctrl+b%
    3. Split the right-hand pane horizontally (stacked): press Ctrl+b"
    4. Navigate to the bottom-right pane using the arrow keys: Ctrl+b↓
    5. Zoom into this pane to focus on it full-screen: press Ctrl+bz
    6. Restore the layout by zooming back out: press Ctrl+bz again.
    7. Cycle through the built-in layouts to see preset options: press Ctrl+bSpacebar multiple times.
    Verification Checklist

    When zoomed in, verify that the status bar displays a *Z next to the window name (e.g., 1:zsh*Z), indicating that a pane is currently maximized.

    6. Copy mode

    Copy mode lets you scroll back through output, search it, and copy text using only the keyboard. tmux ships with emacs-style keys by default, but most people switch to vi-style keys (shown here, and set up in the config section).

    KeysAction
    prefix[Enter copy mode. Now you can scroll and select.
    ←↑↓→ or hjklMove the cursor (vi keys when vi mode is on).
    Ctrl+u Ctrl+dScroll up, or down, half a page.
    / then textSearch forward. ? searches backward, n finds the next match.
    vStart a selection (vi mode).
    yYank the selection and leave copy mode (vi mode).
    prefix]Paste what you copied.
    qQuit copy mode without copying.
    System clipboard

    By default, copied text stays inside tmux. To copy straight to your operating system clipboard, add the tmux-yank plugin covered in the plugins section.

    Lab 06 Scrollback Time Machine

    Navigate scroll history and grab text without touching your mouse.

    1. Generate scroll history in your shell: run history or print a large directory with ls -la /usr/bin
    2. Enter copy mode: press Ctrl+b[ (a cursor will appear in the top-right).
    3. Scroll up and down using the arrow keys, PageUp/PageDown, or vi movement keys (k/j).
    4. Search backward for the command you ran: press ?, type history, and press Enter. Use n to find previous matches.
    5. Press q to exit copy mode and return to your active prompt.
    Verification Checklist

    Upon pressing q, the copy mode indicator in the top-right disappears, and keyboard input is routed back to the command line.

    7. Configuration

    tmux reads ~/.tmux.conf on startup. This is where you make it yours: a friendlier prefix, mouse support, vi keys, a base index that starts at 1, and a cleaner status bar. Create the file and paste the block below.

    # --- prefix: Ctrl+Space, easier than Ctrl+b and rarely clashes ---
unbind C-b
set -g prefix C-Space
bind C-Space send-prefix

# --- quality of life ---
set -g mouse on                 # click panes, drag borders, scroll
set -g base-index 1             # windows start at 1, not 0
setw -g pane-base-index 1       # panes start at 1 too
set -g renumber-windows on      # keep numbers tidy after closing
set -g history-limit 20000       # longer scrollback limit
set -sg escape-time 0           # instant ESC for Neovim/vim
setw -g mode-keys vi             # vi-style keys in copy mode
set -g default-terminal "tmux-256color"
    Reach for the spacebar

    This config sets the prefix to Ctrl+Space. Your thumb already rests on the spacebar, so it is quick to reach with either hand and comfortable to hold while you tap the next key. After you reload, every prefix in this guide is your Ctrl+Space.

    Lab 07 The custom.conf Reboot

    Swap the prefix key and customize quality-of-life behaviors.

    1. Create or open the tmux configuration file in an editor: nano ~/.tmux.conf
    2. Add this rule to change the prefix to Ctrl+Space: 
      unbind C-b
set -g prefix C-Space
bind C-Space send-prefix
    3. Save the file and exit the editor.
    4. Load the changes without quitting tmux. Using the current (default) prefix, press Ctrl+b:, type source-file ~/.tmux.conf, and hit Enter.
    Verification Checklist

    Test the new prefix. Pressing Ctrl+Spacec should now open a new window, while Ctrl+bc no longer does anything.

    8. Plugins

    tmux plugins add useful helpers like clipboard support, session restore, and status line polish. A plugin manager keeps everything tidy and lets you install or update with a single keystroke.

    TPM

    tmux-plugins/tpm

    The standard plugin manager for tmux. Press prefix+I to install and prefix+U to update plugins.

    CLIPBOARD

    tmux-plugins/tmux-yank

    Copy selections from tmux to your system clipboard. Essential for Linux, macOS, WSL, and remote workflows.

    PERSISTENCE

    tmux-plugins/tmux-resurrect

    Save and restore sessions, windows, and panes after rebooting or reloading tmux. Great for long-lived projects.

    Plugin workflow

    Add plugins to ~/.tmux.conf, then reload tmux and press prefix+I. If a plugin breaks your setup, remove it and reload; small changes are easier to recover from.

    # plugin manager and useful plugins
set -g @plugin 'tmux-plugins/tpm'
set -g @plugin 'tmux-plugins/tmux-yank'
set -g @plugin 'tmux-plugins/tmux-resurrect'
run-shell '~/.tmux/plugins/tpm/tpm'

    Lab 08 TPM Quick Start

    Install the plugin manager to download and run community plug-ins.

    1. Clone the Tmux Plugin Manager repository: 
      git clone https://github.com/tmux-plugins/tpm ~/.tmux/plugins/tpm
    2. Open your ~/.tmux.conf and add these configuration lines at the bottom: 
      set -g @plugin 'tmux-plugins/tpm'
set -g @plugin 'tmux-plugins/tmux-yank'
run-shell '~/.tmux/plugins/tpm/tpm'
    3. Source the configuration file: press prefix:, type source-file ~/.tmux.conf, and hit Enter.
    4. Trigger TPM to download and install the plug-ins: press prefixI (capital 'i').
    Verification Checklist

    TPM will open a temporary window showing the installation logs. Once finished, verify that copy mode yank operations now automatically update your system clipboard.

    9. Advanced

    Advanced tmux flows combine scripting, status customization, and session persistence. These patterns keep tmux fast and predictable while letting you automate routine setup.

    KeysAction
    prefix!Break the current pane out into its own window.
    prefix;Toggle to the most recently active pane.
    prefix:Open the tmux command prompt for custom commands.
    prefixxKill the current pane, useful for closing broken shells quickly.
    prefixrReload ~/.tmux.conf after making configuration changes.
    Keep it simple

    Long tmux configs are powerful, but the easiest setup is the one you can read and restore quickly. Start with a few plugins and a small status line before layering more automation.

    Lab 09 The Auto-Dev Environment

    Write a shell script to spin up a multi-pane development workstation automatically.

    1. Create a script file named dev-layout.sh: 
      #!/bin/bash
# Start new session named 'dev' detached
tmux new-session -d -s dev -n editor
# Split the window vertically (left-right)
tmux split-window -h -t dev:editor
# Run htop in the second pane (right side)
tmux send-keys -t dev:editor.2 'htop' Enter
# Attach to the session
tmux attach-session -t dev
    2. Make the script executable: chmod +x dev-layout.sh
    3. Run the script: ./dev-layout.sh
    Verification Checklist

    You should be dropped directly into the newly created dev session, split into two panes, with the right pane actively running htop.
